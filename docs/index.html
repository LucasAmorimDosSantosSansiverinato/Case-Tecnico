Camadas — Clean Architecture

Cada camada é um módulo Maven independente. As dependências de compilação seguem estritamente a hierarquia abaixo:

desafioTecnico-domain Entidades, Value Objects, exceções. Não conhece nenhuma outra camada.
desafioTecnico-application Commands, Queries, Handlers, Ports (interfaces). Conhece apenas domain.
desafioTecnico-infra-data JPA entities, Spring Data repositories, adapter que implementa PersonRepository do domain. Não conhece application.
desafioTecnico-infra-utils ViaCepAddressService, LoginGeneratorService. Conhece domain e application (para implementar as ports). Não conhece webui.
desafioTecnico-webui REST Controllers, DTOs, Exception Handler, Application.java. Conhece apenas application em compile-time. Recebe infra-data e infra-utils via DI em runtime.
A separação física garante que um módulo não pode importar classes de outra camada que não está no seu pom.xml. Qualquer violação é erro de compilação.

CQRS

Commands e Queries são separados em pacotes distintos dentro da camada application. Os handlers são classes simples (sem interface genérica) injetadas via Spring no webui.

BFF — Backend for Frontend

Camada intermediária entre o React e o backend Java. Responsabilidades:

  • Proxy de requisições para o backend (evita CORS direto do browser)
  • Lookup de CEP via ViaCEP e retorno normalizado ao frontend
  • Validação de entrada antes de repassar ao backend
  • Ponto único de controle de CORS

Lógica de Geração do Login

Login: exatamente 7 caracteres · apenas letras minúsculas a–z · sem números · sem espaços · único · derivado do nome da pessoa

Algoritmo (LoginGeneratorService)

1
Normalizar o nome Remove acentos via Normalizer.NFD, converte para minúsculas, elimina caracteres não-alfa. Ex: "João Pedro Lima" → "joao pedro lima"
2
Dividir em partes Split por espaço → ["joao", "pedro", "lima"]
3
Gerar candidatos em ordem de prioridade Estratégia 1: prefixo(firstName, k) + prefixo(sobrenome, 7-k) — variando k de 1 a 7
Estratégia 2: prefixo(sobrenome, k) + prefixo(firstName, 7-k)
Estratégia 3: janela deslizante sobre firstName+sobrenome
Estratégia 4: janela deslizante sobre todos os nomes concatenados
4
Verificar unicidade Compara com todos os logins já cadastrados. Retorna o primeiro candidato disponível.
5
Fallback Se todos os candidatos estiverem ocupados, usa rotação circular sobre os caracteres do nome completo com offset crescente.

Exemplos

NomeLogin geradoEstratégia usada
Maria Silva Souzamariasiprefix(maria,5) + prefix(silva,2)
João Pedro Limajoaopedprefix(joao,4) + prefix(pedro,3)
Ana Paula Ferreiraanapferprefix(ana,3) + prefix(ferreira,4)
O algoritmo garante que nenhum número é inserido. Caso o nome seja muito curto para gerar 7 letras distintas, a rotação circular reutiliza os próprios caracteres do nome em ordem diferente.

Persistência

Banco escolhido: PostgreSQL 16 (relacional), rodando via Docker.

Motivação: dados estruturados com unicidade garantida por constraints (UNIQUE em document, email e login), transações ACID e maturidade do driver JDBC.

Schema da tabela persons

CREATE TABLE persons (
    id           UUID         PRIMARY KEY,
    full_name    VARCHAR(255) NOT NULL,
    document     VARCHAR(11)  NOT NULL UNIQUE,
    email        VARCHAR(255) NOT NULL UNIQUE,
    birth_date   DATE         NOT NULL,
    cep          VARCHAR(8)   NOT NULL,
    street       VARCHAR(255) NOT NULL,
    neighborhood VARCHAR(255),
    city         VARCHAR(255) NOT NULL,
    state        VARCHAR(2)   NOT NULL,
    complement   VARCHAR(255),
    number       VARCHAR(20),
    login        VARCHAR(7)   NOT NULL UNIQUE,
    created_at   TIMESTAMP    NOT NULL DEFAULT NOW()
);

Fluxo do formulário

1
Usuário digita o CEP Ao completar 8 dígitos, useCepLookup chama GET /api/address/{cep} no BFF e preenche logradouro, bairro, cidade e UF automaticamente.
2
Validação client-side React Hook Form valida todos os campos antes do submit (CPF, e-mail, data, nome sem acentos).
3
Submit → BFF → Backend POST para /api/persons no BFF, que repassa ao backend.
4
Exibição do resultado Componente SuccessResult exibe o login gerado em destaque junto com todos os dados cadastrados.

FLYWAY_USER=postgres \
FLYWAY_PASSWORD=postgres \
mvn flyway:migrate

Como Rodar

Opção 1 — Docker Compose (recomendado)

# Na raiz do projeto
docker compose up --build

Aguarde todos os serviços subirem. Frontend pode ser iniciado separadamente:

cd frontend
npm install
npm run dev     # http://localhost:5173

Opção 2 — Manual

# 1. Banco
docker run -d --name pg -e POSTGRES_DB=desafiotecnico \
  -e POSTGRES_USER=postgres -e POSTGRES_PASSWORD=postgres \
  -p 5432:5432 postgres:16

# 2. Migrations
cd migrations
mvn flyway:migrate

# 3. Backend
cd backend
mvn clean package -DskipTests
java -jar desafioTecnico-webui/target/*.jar

# 4. BFF
cd bff
mvn clean package -DskipTests
java -jar target/*.jar

# 5. Frontend
cd frontend
npm install && npm run dev
